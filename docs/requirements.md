# Требования к системе автоматической генерации технической документации

Документ определяет функциональные и нефункциональные требования к системе AnDocGen, сценарии использования, типы входных данных и форматы выходной документации.

## 1. Функциональные требования

### 1.1. Анализ проекта

- **FR-1.1.** Система должна принимать на вход каталог проекта.
- **FR-1.2.** Система должна определять структуру проекта: список файлов, модули, классы, функции, методы.
- **FR-1.3.** Система должна выполнять синтаксический анализ исходного кода поддерживаемых языков программирования.
- **FR-1.4.** Система должна извлекать из кода имена программных сущностей, сигнатуры, параметры и их типы, возвращаемые значения, комментарии, существующую документацию, связи между сущностями и модулями.

### 1.2. Генерация документации

- **FR-2.1.** Система должна генерировать описание для каждой функции: назначение, параметры, возвращаемое значение, возможные исключения или ошибки.
- **FR-2.2.** Система должна генерировать описание для каждого класса: назначение, основные поля и методы.
- **FR-2.3.** Система должна формировать документацию для модуля: краткое назначение, перечень основных классов и функций.
- **FR-2.4.** Система должна использовать контекст проекта при генерации: вызываемые функции, импортируемые модули, имена переменных, существующие комментарии.
- **FR-2.5.** Система должна поддерживать генерацию документации только для сущностей без документации или требующих обновления.
- **FR-2.6.** Система должна поддерживать повторную генерацию документации после изменения исходного кода.

### 1.3. Проверка корректности результата

- **FR-3.1.** Система должна проверять, что все параметры функции отражены в документации.
- **FR-3.2.** Система должна проверять соответствие описанных параметров фактической сигнатуре функции.
- **FR-3.3.** Система должна проверять наличие описания возвращаемого значения для функций, возвращающих результат.
- **FR-3.4.** Система должна обнаруживать упоминания несуществующих параметров, типов или функций в сгенерированной документации.
- **FR-3.5.** Система должна выводить предупреждения при неполном описании, противоречиях между кодом и документацией, невозможности определить назначение функции.

## 2. Нефункциональные требования

### 2.1. Производительность

- **NFR-1.1.** Время генерации документации для одного файла объёмом до 500 строк не должно превышать 10 секунд.
- **NFR-1.2.** Время генерации документации для проекта, содержащего до 50 файлов, не должно превышать 5 минут.
- **NFR-1.3.** Система должна поддерживать повторную генерацию только изменённых файлов без полной обработки проекта.

### 2.2. Масштабируемость

- **NFR-2.1.** Система должна быть готова к добавлению новых языков программирования, форматов документации и модулей проверки.
- **NFR-2.2.** Система должна поддерживать обработку проектов, состоящих как минимум из 50 файлов.

### 2.3. Удобство использования

- **NFR-3.1.** Интерфейс работы с системой должен отображать список обработанных файлов, возникшие ошибки, предупреждения и краткое описание итогового результата.
- **NFR-3.2.** Сообщения об ошибках должны содержать указание проблемного файла, класса или функции.
- **NFR-3.3.** Система должна поддерживать внешний файл конфигурации для управления параметрами обработки.

### 2.4. Входные и выходные данные

- **NFR-4.1.** Система должна принимать исходный код в текстовом формате.
- **NFR-4.2.** Система должна сохранять выходную документацию в текстовом формате.
- **NFR-4.3.** Система должна поддерживать экспорт документации в формат Markdown.

## 3. Сценарии использования

### UC-1. Генерация документации для проекта

1. Пользователь указывает каталог проекта и конфигурацию обработки.
2. Система определяет структуру проекта и находит исходные файлы согласно правилам отбора.
3. Система последовательно обрабатывает файлы: извлечение структуры, формирование контекста, генерация документации.
4. Система формирует документацию для каждого модуля, класса и функции.
5. Система сохраняет результаты в выходной каталог и предоставляет сводку по итогам обработки.

### UC-2. Повторная генерация после изменения кода

1. Пользователь изменяет исходный код.
2. Пользователь повторно запускает систему с включённым инкрементальным режимом.
3. Система определяет изменённые файлы.
4. Система повторно генерирует документацию только для изменённых элементов.
5. Система обновляет итоговые документы в выходном каталоге.

### UC-3. Обработка без внешней языковой модели

1. Пользователь выбирает в конфигурации режим генерации, не требующий обращения к внешним сервисам.
2. Система формирует документацию на основе извлечённых метаданных.
3. Пользователь проверяет корректность конвейера обработки без подключения к внешним API.

## 4. Типы входных данных

### 4.1. Поддерживаемые языки

- Система должна поддерживать исходный код на языках **Python** и **Java**.
- Система должна быть расширяема для подключения других языков программирования без изменения остальных этапов конвейера.

### 4.2. Структура входных данных

- Каталог проекта с древовидной структурой файлов и подкаталогов.
- Текстовые файлы исходного кода.
- Комментарии и документирующие блоки, предусмотренные синтаксисом соответствующего языка.
- Статические описания типов и сигнатур, если они присутствуют в исходном коде.

### 4.3. Зависимости между файлами

Система должна определять:

- связи между модулями через механизмы импорта и включения;
- вызовы функций и методов между сущностями проекта;
- иерархию типов и вложенность модулей.

## 5. Форматы выходной документации

### 5.1. Основной формат

- Выходная документация сохраняется в текстовом формате.
- Система должна поддерживать экспорт в Markdown.

### 5.2. Структура выходного документа

Для каждого модуля формируется документ, содержащий:

1. Заголовок проекта и путь к модулю.
2. Краткое описание назначения модуля.
3. Раздел с описанием классов — назначение, поля и методы.
4. Раздел с описанием функций верхнего уровня.

### 5.3. Расположение результатов

- Документация сохраняется в каталог, заданный пользователем в конфигурации.
- Структура выходных файлов должна соответствовать структуре исходного проекта.
