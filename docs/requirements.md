# Требования к системе автоматической генерации технической документации

Документ определяет функциональные и нефункциональные требования к системе AnDocGen, сценарии использования, типы входных данных и форматы выходной документации.

## 1. Функциональные требования

### 1.1. Анализ проекта

- **FR-1.1.** Система должна принимать на вход каталог проекта.
- **FR-1.2.** Система должна определять структуру проекта: список файлов, модули, классы, функции, методы.
- **FR-1.3.** Система должна выполнять синтаксический анализ исходного кода поддерживаемых языков программирования.
- **FR-1.4.** Система должна извлекать из кода имена программных сущностей, сигнатуры, параметры и их типы, возвращаемые значения, комментарии, существующую документацию, связи между сущностями и модулями, а также цикломатическую сложность функций и методов.
- **FR-1.5.** При ошибке разбора отдельного файла система должна продолжать обработку остальных файлов и фиксировать ошибку в отчёте.

### 1.2. Генерация документации

- **FR-2.1.** Система должна генерировать описание для каждой функции: назначение, параметры, возвращаемое значение, возможные исключения или ошибки.
- **FR-2.2.** Система должна генерировать описание для каждого класса: назначение, основные поля и методы.
- **FR-2.3.** Система должна формировать документацию для модуля: краткое назначение, перечень основных классов и функций.
- **FR-2.4.** Система должна использовать контекст проекта при генерации: вызываемые функции, импортируемые модули, имена переменных, существующие комментарии.
- **FR-2.5.** Система должна поддерживать повторную генерацию документации после изменения исходного кода.
- **FR-2.6.** При повторной генерации система должна передавать языковой модели актуальный исходный код сущности и ранее сгенерированную для неё выходную документацию.
- **FR-2.7.** Система должна поддерживать генерацию документации для отдельного модуля или подмножества файлов проекта.
- **FR-2.8.** Система должна позволять исключать из обработки отдельные файлы, каталоги и пути по шаблонам.
- **FR-2.9.** Система должна формировать сводную документацию проекта: название, общее назначение, перечень модулей.
- **FR-2.10.** Система должна позволять через конфигурацию задавать язык генерируемой документации независимо от языка исходного кода.

### 1.3. Проверка корректности результата

- **FR-3.1.** Система должна проверять, что все параметры функции отражены в документации.
- **FR-3.2.** Система должна проверять соответствие описанных параметров фактической сигнатуре функции.
- **FR-3.3.** Система должна проверять наличие описания возвращаемого значения для функций, возвращающих результат.
- **FR-3.4.** Система должна обнаруживать упоминания несуществующих параметров, типов или функций в сгенерированной документации.
- **FR-3.5.** Система должна выводить предупреждения при неполном описании, противоречиях между кодом и документацией, невозможности определить назначение функции.

## 2. Нефункциональные требования

### 2.1. Производительность

- **NFR-1.1.** Время генерации документации для одной программной сущности не регламентируется. Фактическое время зависит от провайдера и скорости языковой модели.
- **NFR-1.2.** Система должна обрабатывать проекты объёмом до 50 исходных файлов; суммарное время зависит от числа сущностей, их сложности и провайдера генерации.
- **NFR-1.3.** При повторной генерации с включённым инкрементальным режимом обрабатываются только изменённые файлы, для каждого такого файла выполняется полная перегенерация всех его сущностей.

### 2.2. Масштабируемость

- **NFR-2.1.** Система должна быть готова к добавлению новых языков программирования, форматов документации и модулей проверки.
- **NFR-2.2.** Система должна поддерживать обработку проектов, состоящих как минимум из 50 файлов.

### 2.3. Удобство использования

- **NFR-3.1.** Система должна формировать три уровня отчётности прогона: краткая сводка, детальный отчёт, файл с полной трассировкой выполнения.
- **NFR-3.2.** Сообщения об ошибках должны содержать указание проблемного файла, класса или функции.
- **NFR-3.3.** Система должна поддерживать внешний файл конфигурации для управления параметрами обработки.
- **NFR-3.4.** Учётные данные для внешних сервисов не должны храниться в исходном коде анализируемого проекта и не должны попадать в выходные документы и пользовательские отчёты.

### 2.4. Входные и выходные данные

- **NFR-4.1.** Система должна принимать исходный код в текстовом формате.
- **NFR-4.2.** Система должна сохранять выходную документацию в текстовом формате.
- **NFR-4.3.** Система должна поддерживать экспорт документации в формат Markdown.

### 2.5. Качество документации

- **NFR-5.1.** Документация должна быть полной: описывать назначение, параметры, возвращаемое значение и значимые побочные эффекты там, где их можно установить из кода.
- **NFR-5.2.** Документация должна быть структурированной: единообразные заголовки, списки, разделы для однотипных сущностей.
- **NFR-5.3.** Документация должна быть читаемой: связный текст, без избыточного дублирования сигнатуры.
- **NFR-5.4.** Документация должна быть согласованной: в рамках одного проекта соблюдается единая терминология, стиль и язык описания.
- **NFR-5.5.** Выходные документы должны обеспечивать навигацию между связанными сущностями и модулями; способ навигации зависит от формата вывода.

### 2.6. Надёжность и обработка ошибок

- **NFR-6.1.** Система должна различать ошибки анализа, ошибки генерации и замечания валидации в итоговом отчёте.
- **NFR-6.2.** При недоступности языковой модели система должна прерывать генерацию и сообщать об ошибке.

## 3. Сценарии использования

### UC-1. Генерация документации для проекта

1. Пользователь указывает каталог проекта и конфигурацию обработки.
2. Система определяет структуру проекта и находит исходные файлы согласно правилам отбора.
3. Система последовательно обрабатывает файлы: извлечение структуры, формирование контекста, генерация документации.
4. Система формирует документацию для каждого модуля, класса и функции, а также сводную документацию проекта.
5. Система сохраняет результаты в выходной каталог и предоставляет сводку по итогам обработки.

### UC-2. Повторная генерация после изменения кода

1. Пользователь изменяет исходный код.
2. Пользователь повторно запускает систему с включённым инкрементальным режимом.
3. Система определяет изменённые файлы по контрольным суммам.
4. Система пересобирает модель и перегенерирует документацию для всех сущностей каждого изменённого файла.
5. Система обновляет итоговые документы и сводный файл проекта в выходном каталоге.

### UC-3. Обработка без внешней языковой модели

1. Пользователь выбирает в конфигурации провайдер-заглушку, не требующий обращения к внешним сервисам.
2. Система формирует документацию на основе шаблонов и извлечённых метаданных.
3. Пользователь проверяет корректность заданных им настроек обработки.

### UC-4. Выборочная генерация для части проекта

1. Пользователь указывает каталог проекта и правила отбора: конкретный модуль, подмножество файлов или исключённые каталоги и паттерны.
2. Система обрабатывает только отобранные файлы.
3. Система формирует документацию для выбранных модулей и предоставляет сводку по итогам обработки.

## 4. Типы входных данных

Система должна поддерживать исходный код на языках **Python 3.10+** и **Java 17+**. Система должна быть расширяема для подключения других языков программирования без изменения остальных этапов обработки.

Входные данные представляют из себя:

- Каталог проекта с древовидной структурой файлов и каталогов.
- Текстовые файлы исходного кода.
- Комментарии и документирующие блоки, предусмотренные синтаксисом соответствующего языка (docstring, Javadoc и аналоги).
- Статические описания типов и сигнатур, если они присутствуют в исходном коде.

Система должна определять связи между модулями через механизмы импорта и включения, вызовы функций и методов между сущностями проекта, иерархию типов и вложенность модулей.

## 5. Форматы выходной документации

Выходная документация сохраняется в текстовом формате.

### 5.1. Сводный файл проекта

Формируется один обзорный документ (в корне выходного каталога), содержащий:

1. Название проекта.
2. Общее назначение (из конфигурации, README и метаданных сборки проекта).
3. Перечень модулей с относительными ссылками на их документацию.

### 5.2. Документы модулей

Для каждого модуля формируется отдельный файл; структура выходных путей повторяет структуру исходного проекта.

Каждый документ модуля содержит:

1. Заголовок проекта и путь к модулю.
2. Краткое описание назначения модуля.
3. Раздел с описанием классов — назначение, поля и методы.
4. Раздел с описанием функций верхнего уровня.

Документация сохраняется в каталог, заданный пользователем в конфигурации.

## 6. Ограничения и допущения

- Система выполняет **только статический анализ** исходного кода; runtime-поведение, динамическая загрузка модулей и выполнение кода не анализируются.
- **Граф вызовов** строится с частичным разрешением имён; вызовы через reflection, `getattr`, `eval` и аналоги могут остаться неразрешёнными.
- **Качество** сгенерированной документации зависит от языковой модели и может содержать неточности; качество ограничено этапом валидации и метриками сравнения с эталонами.
- Для сущностей с высокой цикломатической сложностью время генерации не гарантируется.
- **Не поддерживаются** как входные данные: бинарные файлы, проекты без исходного текста, каталоги со cгенерированным кодом (например, интерфейсы для реализации API).
- Поддержка определенных языковых конструкций описана отдельно; такие конструкции не извлекаются или обрабатываются частично.
