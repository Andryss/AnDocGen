# Описание требований к системе автоматической генерации технической документации

## Функциональные требования:

Анализ проекта:

1.  Система должна принимать на вход каталог проекта
2.  Система должна определять структуру проекта (находить список файлов, модули, классы, функции, методы)
3.  Система должна выполнять синтаксический анализ исходного кода
4.  Система должна извлекать из кода имена функций, сигнатуры, параметры и их типы, возвращаемые значения, комментарии и существующие документации, связи между функциями и модулями

Генерация документации:

1.  Система должна генерировать описание для каждой функции, включая назначение функции, описание параметров, описание возвращаемого значения, описание возможных исключений или ошибок
2.  Система должна генерировать описание для каждого класса, включая описание назначения класса, перечень основных полей и методов
3.  Система должна формировать документацию для модуля, включая краткое назначение модуля, перечень основных классов и функций
4.  Система должна использовать контекст проекта при генерации документации, а именно: учитывать вызываемые функции, импортируемые модули, имена переменных и существующие комментарии
5.  Система должна поддерживать возможность генерации документации только для тех сущностей, которые отсутствуют в исходном коде либо требуют обновления.
6.  Система должна поддерживать возможность повторной генерации документации после изменения исходного кода

Проверка корректности результата:

1.  Система должна проверять, что все параметры функции отражены в документации
2.  Система должна проверять соответствие описанных параметров фактической сигнатуре функции
3.  Система должна проверять наличие описания возвращаемого значения для функций, возвращающих результат
4.  Система должна обнаруживать случаи, когда в сгенерированной документации упоминаются несуществующие параметры, типы или функции
5.  Система должна выводить предупреждения при обнаружении неполного описания, противоречий между кодом и документацией, невозможности определить назначение функции

## Нефункциональные требования

Требования к производительности:

1.  Время генерации документации для одного файла объемом до 500 строк не должно превышать 10 секунд.
2.  Время генерации документации для проекта, содержащего до 50 файлов, не должно превышать 5 минут.
3.  Система должна поддерживать повторную генерацию только измененных файлов без полной обработки проекта.

Требования к масштабируемости:

1.  Система должна быть готова к добавлению новых языков программирования, новых форматов документации, дополнительных модулей проверки
2.  Система должна поддерживать обработку проектов, состоящих как минимум из 50 файлов.

Требования к удобству пользователя:

1.  Интерфейс работы с системой должен отображать список обработанных файлов, возникшие ошибки, предупреждения, краткое описание итогового результата
2.  Сообщения об ошибках должны содержать указание проблемного файла, класса или функции

Требования к входным и выходным данным:

1.  Система должна принимать исходный код в текстовом формате.
2.  Система должна сохранять выходную документацию в текстовом формате.
3.  Документация должна поддерживать экспорт в Markdown.

## Сценарии использования системы

Генерация документации для проекта:

1.  Пользователь выбирает каталог проекта.
2.  Система анализирует структуру проекта.
3.  Система последовательно обрабатывает все файлы.
4.  Система формирует документацию для каждого модуля.
5.  Система объединяет результаты в единую директорию.

Повторная генерация после изменения кода:

1.  Пользователь изменяет исходный код.
2.  Пользователь повторно запускает систему.
3.  Система определяет измененные файлы.
4.  Система повторно генерирует документацию только для измененных элементов.
5.  Система обновляет итоговые документы.

## Типы входных данных

В качестве входных данных система должна поддерживать исходных код проектов, написанных на языках Java и Python. Соответственно система должна поддерживать файлы с исходным кодом этих языков программирования.

Система должна определять дерево каталогов проекта и зависимости между файлами.

Также система должна находить и определять комментарии в существующем коде, поддерживаемые определенными языками программирования.

## Форматы выходной документации

Система должна поддерживать выходную документацию в текстовом формате Markdown.

Выходные документы должны включать: заголовок проекта, список модулей, разделы для классов, разделы для функций.
