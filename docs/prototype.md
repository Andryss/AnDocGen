# Описание прототипа AnDocGen

Документ описывает программную реализацию прототипа системы автоматической генерации технической документации: технологический стек, структуру кода, реализованные модули, примеры работы и ограничения текущей версии. Прототип реализует полный конвейер для исходного кода Python и запускается консольной командой `andocgen`.

## 1. Структура программной реализации

### 1.1. Технологический стек

- Язык реализации — **Python 3.11 и новее** — удобная работа с HTTP API языковых моделей, встроенный синтаксический разбор, единый стек с документируемым Python на текущем этапе
- Разбор документируемого кода — **модуль `ast`** (синтаксис Python 3.10) — стандартное средство языка, не требует внешнего парсера
- Менеджер зависимостей — **pip** — стандартная установка пакета и библиотек в виртуальное окружение
- Описание проекта — **файл `pyproject.toml`** — единый формат метаданных, зависимостей и точки входа
- Сборка пакета — **setuptools** — распространённый инструмент упаковки; исходники в каталоге `src/andocgen/`
- Тестирование — **pytest** — простой запуск модульных и интеграционных тестов

Основные библиотеки:

- Консольный интерфейс — **typer** — объявление команды `andocgen`, аргументов и параметров
- Вывод в консоль — **rich** — форматирование текста и индикатор прогресса прогона
- Конфигурация — **pyyaml** — чтение параметров из файла YAML
- Обращение к провайдерам генерации — **httpx** — HTTP-запросы к Ollama и внешним API, совместимым с OpenAI

### 1.2. Расположение кода

Исходный код прототипа расположен в каталоге `src/andocgen/`. Точка входа для пользователя — модуль командной строки. Оркестратор конвейера связывает все этапы обработки в единый рабочий процесс. Загрузка параметров выполняется отдельным модулем конфигурации.

### 1.3. Дерево каталогов

| Каталог | Назначение |
|---------|------------|
| `scanner/` | обнаружение исходных файлов |
| `parser/` | синтаксический разбор |
| `call_graph/` | построение графа вызовов и порядка сущностей |
| `context/` | сборка контекста сущности и запроса к модели |
| `generator/` | генерация, разбор секций ответа, форматирование |
| `llm/` | провайдеры генерации текста |
| `validator/` | проверка сгенерированной документации |
| `output/` | запись документов, кэш, загрузка предыдущей документации |
| `reporting/` | сводка, детальный отчёт, трассировка, прогресс |
| `models/` | общие структуры данных между этапами |
| `i18n/` | подписи секций вывода на языке документации |

Отдельно в корне пакета: модуль конфигурации, оркестратор конвейера, интерфейс командной строки.

### 1.4. Единый шаблон модуля

Каждый этап конвейера оформлен одинаково: объявление интерфейса, фабрика выбора реализации по параметрам конфигурации, каталог с конкретными реализациями.

### 1.5. Интеграция конвейера

Оркестратор связывает все модули в единый рабочий процесс:

1. Обнаружение файлов в каталоге проекта.
2. Разбор каждого файла; при включённом инкрементальном режиме неизменённые файлы пропускаются.
3. Сборка модели проекта.
4. Построение графа вызовов.
5. Формирование контекстов сущностей и упорядочивание по зависимостям.
6. Генерация документационных блоков.
7. Проверка сгенерированной документации.
8. Запись документов и обновление кэша контрольных сумм.
9. Формирование отчётов прогона.

Ошибка разбора отдельного файла фиксируется в отчёте; обработка остальных файлов продолжается.

### 1.6. Исполняемый файл

После установки пакета в виртуальное окружение командой `pip install -e ".[dev]"` в системе появляется консольная команда `andocgen`. Она зарегистрирована как точка входа пакета в файле описания проекта `pyproject.toml` в корне репозитория.

### 2.2. Обнаружение

Рекурсивный обход каталога проекта: отбор файлов по расширению, исключение служебных каталогов, вычисление контрольной суммы SHA-256 для инкрементального режима.

### 2.3. Извлечение

Синтаксический разбор исходного кода Python через модуль `ast`. Извлекаются функции, классы, импорты, вызовы, цикломатическая сложность и декораторы. Поддерживается синтаксис Python 3.10.

### 2.4. Граф вызовов

Построение статического графа вызовов между сущностями проекта, разрешение имён с учётом импортов, топологическое упорядочивание сущностей перед генерацией.

### 2.5. Подготовка контекста

Для каждой сущности формируется контекст: сигнатура, исходный код, импорты, документация вызываемых сущностей, фрагмент описания проекта. Отдельный компонент собирает системное и пользовательское сообщения для провайдера генерации. Загрузчик метаданных читает название и описание проекта из конфигурации и файла описания проекта в каталоге.

### 2.6. Генерация

Оркестрация генерации по сущностям: запрос к провайдеру, разбор секционного ответа, форматирование для вывода. Сущности обрабатываются волнами с учётом зависимостей; при числе рабочих потоков больше одного запросы к модели выполняются параллельно внутри волны. Во время генерации выполняются предварительные проверки с повторными попытками при блокирующих замечаниях.

Доступны три провайдера: заглушка (детерминированный ответ без внешних сервисов), локальный сервер Ollama, внешний API, совместимый с OpenAI.

### 2.7. Валидация

Проверка сгенерированных блоков относительно структурной модели. Группы проверок (полнота, согласованность с кодом, структура ответа, согласованность представлений, качество текста) включаются и отключаются параметрами конфигурации.

### 2.8. Вывод

Запись сводного файла проекта и документов модулей в формате Markdown с сохранением структуры исходного каталога. Кэш хранит контрольные суммы обработанных файлов. При повторной генерации из ранее записанных документов извлекаются блоки соответствующих сущностей.

### 2.9. Отчётность

Формирование краткой сводки, детального отчёта в формате JSON и файла трассировки. В консоли отображаются этапы обработки и итоговая статистика прогона.

### 2.10. Общие структуры данных

Пакет `models/` содержит классы данных, передаваемые между этапами: модель проекта, модуль, контекст сущности, документационный блок, результат прогона, ошибки разбора и генерации.

### 2.11. Оркестрация и командная строка

Оркестратор создаёт компоненты через фабрики, выполняет этапы в нужном порядке и возвращает итог прогона. Командная строка принимает путь к каталогу проекта и необязательный файл конфигурации, затем запускает конвейер.

## 3. Примеры входных данных и сгенерированных документов

Примеры сгенерированы провайдером **Ollama** (модель `qwen2.5-coder:7b`, локальный сервер `http://localhost:11434`). Для каждого примера в каталоге проекта лежит файл `config.ollama.yaml` с параметрами генерации; результат сохранён в подкаталоге `generated_docs/` рядом с исходниками.

Запуск (из корня репозитория, после установки пакета и запуска Ollama):

```bash
andocgen examples/mini_calculator --config examples/mini_calculator/config.ollama.yaml
andocgen examples/mini_api --config examples/mini_api/config.ollama.yaml
andocgen examples/mini_library --config examples/mini_library/config.ollama.yaml
```

### 3.1. Калькулятор (mini_calculator)

Входной проект — каталог `examples/mini_calculator/`. Исходный файл `calculator.py`:

```python
"""Simple calculator module."""

from typing import Optional


def add(a: float, b: float) -> float:
    """Add two numbers."""
    return a + b


def divide(a: float, b: float) -> Optional[float]:
    if b == 0:
        return None
    return a / b


class Calculator:
    """Performs basic arithmetic operations."""

    def __init__(self, precision: int = 2) -> None:
        self.precision = precision

    def multiply(self, x: int, y: int) -> int:
        return x * y

    def format_result(self, value: float) -> str:
        return f"{value:.{self.precision}f}"
```

Прогон: 7 сущностей, ~45 с. Выходные файлы — `examples/mini_calculator/generated_docs/README.md` и `examples/mini_calculator/generated_docs/calculator.py.md`.

Фрагмент сгенерированной документации для функции `add` (полный текст — в `examples/mini_calculator/generated_docs/calculator.py.md`):

```markdown
### `def add(a: float, b: float) -> float`

Складывает два числа и возвращает результат.

**Параметры:**

- `a` (`float`) — первое слагаемое
- `b` (`float`) — второе слагаемое

**Возвращаемое значение:**

- `float` — сумма двух чисел
```

### 3.2. API-пример (mini_api)

Мультимодульный пример — каталог `examples/mini_api/`. Модули `handlers.py` и `storage.py` связаны вызовами между файлами; при генерации в контекст сущности попадает документация вызываемых функций из другого модуля.

Прогон: 13 сущностей в 3 файлах, ~59 с. Результат — каталог `examples/mini_api/generated_docs/`:

| Файл | Содержание |
|------|------------|
| `README.md` | сводный индекс проекта |
| `handlers.py.md` | классы `ApiResponse`, `UserHandler` и их методы |
| `storage.py.md` | класс `InMemoryStorage`, функция `normalize_email` |
| `__init__.py.md` | описание пакета |

Фрагмент документации метода `create_user` — видна связь с функцией из соседнего модуля:

```markdown
##### `create_user(name: str, email: str) -> ApiResponse`

Создает нового пользователя и возвращает ответ с информацией о созданном пользователе.

**Параметры:**

- `name` (`str`) — имя пользователя
- `email` (`str`) — адрес электронной почты пользователя

**Возвращаемое значение:**

- `ApiResponse` — ответ, содержащий статус 201 и ID созданного пользователя

**Смотрите также:**

- `normalize_email`
```

### 3.3. Библиотека заказов (mini_library)

Расширенный мультимодульный пример — каталог `examples/mini_library/`. Проект включает модели, хранилище, бизнес-логику, плагины экспорта, утилиты и консольный интерфейс; модули связаны импортами и вызовами между пакетами (`services`, `storage`, `utils`, `plugins`).

Прогон: 40 сущностей в 12 файлах, ~4 мин. Результат — каталог `examples/mini_library/generated_docs/` со сводным `README.md`, документами модулей и вложенными индексами пакетов `plugins/` и `utils/`.

Фрагмент документации метода `quote_total` — видна связь с функцией форматирования из подпакета `utils`:

```markdown
##### `quote_total(order_id: int) -> str`

Вычисляет общий счет для заказа и возвращает его строковое представление.

**Параметры:**

- `order_id` (`int`) — уникальный идентификатор заказа

**Возвращаемое значение:**

- `str` — общий счет заказа, отформатированный с учетом валюты

**Смотрите также:**

- `format_price` — Форматирует сумму денег с символом валюты.
```

## 4. Ограничения реализации и запуск

### 4.1. Ограничения прототипа

В прототипе не реализованы:

- **поддержка Java** — извлечение структуры из исходников на Java;
- **форматы вывода кроме Markdown** — HTML, PDF и другие варианты записи документации;
- **сравнение с эталоном** — автоматические метрики качества и сопоставление с существующей документацией;
- **выборочная генерация** — запуск для отдельного модуля или заданного набора файлов (обрабатывается весь каталог проекта с учётом исключений в конфигурации);
- **веб- или графический интерфейс** — взаимодействие только через консольную команду.

### 4.2. Установка и конфигурация

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
```

Пример конфигурации можно найти в корне репозитория в файле `config.example.yaml`. В конфигурации задаются разделы: проект, обнаружение, извлечение, граф вызовов, контекст, генерация, валидация, вывод, отчётность.

Запуск генерации для примера калькулятора:

```bash
andocgen examples/mini_calculator --config config.example.yaml
```

### 4.3. Выходные артефакты

После успешного прогона в каталоге вывода, указанном в конфигурации, формируются:

- сводный файл проекта (`README.md`);
- документы модулей в формате Markdown (структура каталогов повторяет исходный проект);
- кэш контрольных сумм для инкрементального режима;
- каталог отчётов: краткая сводка, детальный отчёт, файл трассировки.
