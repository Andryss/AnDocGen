# Первичная апробация системы AnDocGen

Документ содержит результаты тестовых запусков прототипа на контролируемых примерах и внешнем open-source проекте: примеры успешной и неуспешной генерации, анализ причин ошибок, ограничения подхода и направления дальнейшего развития.

## 1. Методика апробации

Апробация выполнена на локальной машине с установленным прототипом AnDocGen и сервером Ollama. Все прогоны использовали модель `qwen2.5-coder:7b`; язык генерируемой документации — русский.

### 1.1. Условия прогонов

- среда — Python 3.11+, macOS;
- провайдер генерации — Ollama (`http://localhost:11434`), модель `qwen2.5-coder:7b`;
- язык документации — русский (параметр `language: ru` в файле конфигурации).

### 1.2. Набор тестовых проектов

Использовались два уровня входных данных:

- **контролируемые примеры** из каталога `examples/` — воспроизводимые проекты с известной структурой;
- **внешний open-source проект** — библиотека python-qrcode как git-подмодуль в `eval/python-qrcode/python-qrcode/`.

Конкретные проекты:

- `examples/mini_calculator` — один модуль с функциями и классом;
- `examples/mini_api` — несколько модулей с вызовами между файлами;
- `examples/mini_library` — мультимодульный проект (~12 файлов): сервисы, CLI, плагины;
- `eval/python-qrcode/python-qrcode` — библиотека [lincolnloop/python-qrcode](https://github.com/lincolnloop/python-qrcode), 25 исходных модулей (подмодуль git).

### 1.3. Команды запуска

```bash
pip install -e .

andocgen examples/mini_calculator --config examples/mini_calculator/config.ollama.yaml
andocgen examples/mini_api --config examples/mini_api/config.ollama.yaml
andocgen examples/mini_library --config examples/mini_library/config.ollama.yaml

git submodule update --init eval/python-qrcode/python-qrcode

andocgen eval/python-qrcode/python-qrcode/qrcode --config eval/python-qrcode/python-qrcode.config.yaml
```

Результаты сохраняются в каталогах `generated_docs/` рядом с исходниками (для примеров из `examples/`) или в `eval/python-qrcode/generated_docs/`.

### 1.4. Критерии оценки прогона

Прогон считается успешным, если:

- нет ошибок синтаксического разбора исходников;
- нет ошибок генерации (каждая сущность попала в выходной документ);
- предупреждения валидации допустимы в единичных случаях (неполное описание, язык, высокая сложность);
- сформированы сводный `README.md` и документы модулей с сохранением структуры проекта.

## 2. Примеры успешной генерации

Для каждого примера ниже указаны каталог исходного кода и каталог с результатами прогона (`generated_docs/`). Структура выходных файлов повторяет структуру исходного проекта.

### 2.1. Калькулятор (`examples/mini_calculator`)

**Исходники:** `examples/mini_calculator/`

**Результаты прогона:** `examples/mini_calculator/generated_docs/` — сводный `README.md`, документ `calculator.py.md`, логи в `logs/`.

**Вход:** один модуль `calculator.py` с функциями `add`, `divide` и классом `Calculator`.

**Прогон:** 7 сущностей, 1 файл, ~45 с. Ошибок и предупреждений нет.

**Фрагмент сгенерированной документации для функции `add`:**

```markdown
### `def add(a: float, b: float) -> float`

Складывает два числа и возвращает результат.

**Параметры:**

- `a` (`float`) — первое слагаемое
- `b` (`float`) — второе слагаемое

**Возвращаемое значение:**

- `float` — сумма двух чисел
```

**Оценка:** успешно. Парсер извлёк сигнатуры и docstring; модель сформировала описание на русском языке с параметрами и возвращаемым значением; валидатор замечаний не выдал.

### 2.2. API-пример (`examples/mini_api`)

**Исходники:** `examples/mini_api/`

**Результаты прогона:** `examples/mini_api/generated_docs/` — `README.md`, `handlers.py.md`, `storage.py.md`, `__init__.py.md`, логи в `logs/`.

**Вход:** три модуля — `handlers.py`, `storage.py`, `__init__.py`.

**Прогон:** 13 сущностей, 3 файла, ~59 с. Одно предупреждение валидации.

**Фрагмент документации метода `create_user` — видна связь с функцией из соседнего модуля:**

```markdown
##### `create_user(name: str, email: str) -> ApiResponse`

Создает нового пользователя и возвращает ответ с информацией о созданном пользователе.

**Параметры:**

- `name` (`str`) — имя пользователя
- `email` (`str`) — адрес электронной почты пользователя

**Возвращаемое значение:**

- `ApiResponse` — ответ, содержащий статус 201 и ID созданного пользователя

**Смотрите также:**

- `normalize_email`
```

**Оценка:** успешно. Мультимодульный проект обработан целиком; структура каталогов сохранена; в контекст генерации попала документация вызываемой функции из другого файла.

### 2.3. Библиотека заказов (`examples/mini_library`)

**Исходники:** `examples/mini_library/`

**Результаты прогона:** `examples/mini_library/generated_docs/` — сводный `README.md`, документы модулей в корне и в подкаталогах `plugins/`, `utils/`, логи в `logs/`.

**Вход:** 12 файлов — модели, хранилище, сервисы, CLI, плагины, утилиты.

**Прогон:** 40 сущностей, ~4 мин. Три предупреждения валидации (некорректные примеры кода в описаниях).

**Фрагмент метода `quote_total` — связь с функцией форматирования из подпакета `utils`:**

```markdown
##### `quote_total(order_id: int) -> str`

Вычисляет общий счет для заказа и возвращает его строковое представление.

**Параметры:**

- `order_id` (`int`) — уникальный идентификатор заказа

**Возвращаемое значение:**

- `str` — общий счет заказа, отформатированный с учетом валюты

**Смотрите также:**

- `format_price` — Форматирует сумму денег с символом валюты.
```

**Оценка:** успешно. Конвейер масштабируется на проект с несколькими пакетами; документация согласована по структуре; предупреждения касаются только примеров кода, а не пропуска сущностей.

### 2.4. Внешний проект python-qrcode

**Исходники:** `eval/python-qrcode/python-qrcode/qrcode/` (git-подмодуль upstream-репозитория)

**Результаты прогона:** `eval/python-qrcode/generated_docs/` — сводный `README.md`, 25 документов модулей (в том числе в подкаталоге `image/`), логи в `logs/`.

**Вход:** 25 Python-модулей (тесты исключены конфигурацией).

**Прогон:** ~29 мин, 1 ошибка генерации, 9 предупреждений валидации. Сформировано 26 выходных файлов.

**Что сгенерировано успешно:**

1. **Публичный API** — модуль `__init__.py.md` перечисляет константы коррекции ошибок, класс `QRCode`, функции `make` и `run_example`.
2. **CLI** — `console_scripts.py.md` содержит описание `main()`, параметров командной строки, исключений и примеров вызова.
3. **Утилиты** — `util.py.md`, `LUT.py.md` и другие вспомогательные модули получили описания классов и функций с параметрами на русском языке.
4. **Подсистема изображений** — модули `image/` задокументированы с полями, наследованием и методами.

**Фрагмент из `console_scripts.py.md`:**

```markdown
### `def get_factory(module: str) -> type[BaseImage]`

Возвращает тип изображения-фабрики по полному пути модуля.

**Параметры:**

- `module` (`str`) — полный путь к модулю, содержащему фабрику изображений

**Возвращаемое значение:**

- `type[BaseImage]` — тип фабрики изображений

**Исключения:**

- `ValueError` — если указан неполный путь к модулю
```

**Оценка:** в основном успешно. Парсер обработал все 25 файлов без ошибок; большинство сущностей получили осмысленные описания. Критичный пробел — отсутствие документации класса `QRCode` (см. §3.1).

## 3. Примеры неуспешной генерации

### 3.1. Несоблюдение секционного формата (python-qrcode)

**Сущность:** класс `QRCode` в модуле `main.py`.

**Проблема:** модель вернула ответ без обязательных секций `Fields`, `Inheritance`, `Methods overview`. Сущность не попала в итоговый файл `main.py.md`; в документации проекта отсутствует описание центрального класса библиотеки.

**Сообщение об ошибке:** `Missing required sections: Fields, Inheritance, Methods overview`.

**Следствие:** пользователь видит функции и вспомогательные классы в `main.py.md`, но не видит документацию основного API.

### 3.2. Смешение языков (python-qrcode)

**Сущность:** функция `run_example` в `__init__.py`.

**Проблема:** при настройке `language: ru` краткое описание сгенерировано на английском языке: «Builds an example QR Code and displays it.» Аналогичное предупреждение выдано для метода `BaseImage.pixel_box` в `image/base.py`.

**Обнаружение:** валидатор — «Summary appears to be in a different language than configured (ru)».

### 3.3. Некорректные или неполные описания (mini_library, python-qrcode)

**mini_library** — предупреждения валидации примеров кода:

- в методе `Order.total` пример ссылается на поля `price` и `quantity`, которых нет в модели `Order`;
- в методе `OrderService.summarize` пример создаёт объект без обязательного аргумента `storage`.

**python-qrcode** — предупреждения о слишком коротких описаниях (`Summary appears too short`) для ряда методов в подсистеме `image/`; предупреждение о высокой цикломатической сложности (41) для функции `_lost_point_level3` в `util.py`.

**Проблема:** текст документации формально присутствует, но примеры могут вводить в заблуждение, а краткие описания не раскрывают назначение методов.

## 4. Анализ причин возникновения ошибок

### 4.1. Ошибка генерации (пропуск сущности)

Для крупного класса `QRCode` модель вернула ответ не в секционном формате, поэтому сущность не попала в итоговую документацию.

**Возможные решения:** повторный запрос с текстом ошибки разбора; сокращение контекста для классов; резервное описание из docstring при неудаче.

### 4.2. Смешение языков

При настройке русского языка отдельные описания (`run_example`, `BaseImage.pixel_box`) сформированы на английском, вероятно под влиянием docstring исходника.

**Возможные решения:** усилить инструкцию о языке в промпте; пост-проверка языка с повторной генерацией; явно указывать целевой язык в контексте сущности.

### 4.3. Некорректные примеры кода

В описаниях `Order.total` и `OrderService.summarize` примеры ссылаются на несуществующие поля или пропускают обязательные аргументы конструктора.

**Возможные решения:** передавать в промпт явный список полей и параметров из модели; проверять примеры валидатором и отклонять или перегенерировать при расхождении.

### 4.4. Слишком краткие описания

Для ряда методов в подсистеме `image/` python-qrcode краткое описание не раскрывает назначение метода.

**Возможные решения:** задать минимальную длину summary в промпте; для служебных методов использовать шаблон с указанием роли в классе; повторная генерация при предупреждении валидатора.

### 4.5. Высокая цикломатическая сложность

Функция `_lost_point_level3` получила предупреждение валидатора из-за сложности 41, что указывает на риск неполного описания ветвлений.

**Возможные решения:** для функций выше порога сложности генерировать только краткое summary; разбивать описание на отдельные блоки по основным веткам; предупреждать пользователя в отчёте прогона.

### 4.6. Общий вывод

Основные сбои связаны с поведением языковой модели (формат, язык, достоверность примеров), а не с ошибками парсера исходного кода. Статический анализ и валидатор выявляют часть проблем постфактум, но не предотвращают их полностью.

## 5. Ограничения разработанного подхода

- **Зависимость от языковой модели** — качество, язык и структура текста определяются провайдером генерации; локальная модель 7B не всегда соблюдает секционный контракт и настройку языка.
- **Ограниченный статический контекст** — система не выполняет семантический анализ тела функций; сложные и динамические конструкции описываются обобщённо или с ошибками в примерах.
- **Хрупкость секционного формата** — при нарушении структуры ответа сущность целиком исключается из выходной документации, а не сохраняется в упрощённом виде.
- **Эвристическая валидация** — проверки выявляют несогласованность с сигнатурой и языком, но не гарантируют семантическую корректность описаний и примеров.
- **Масштабирование по времени** — прогон python-qrcode занял ~29 мин; время растёт линейно с числом сущностей и зависит от скорости локальной модели.
- **Контекст крупных сущностей** — для больших классов и модулей объём передаваемого исходного кода может перегружать модель и ухудшать соблюдение формата ответа.

## 6. Направления дальнейшего развития

- **Устойчивость к ошибкам формата** — повторные запросы с указанием ошибки разбора; упрощённый fallback при неудаче (краткое описание из docstring).
- **Управление контекстом** — сокращение тела исходника для крупных классов; приоритет сигнатуры, docstring и графа вызовов.
- **Контроль языка и достоверности** — усиление инструкций в промпте; пост-проверка языка; фильтрация параметров и полей, отсутствующих в модели.
- **Расширение валидации** — семантические проверки примеров кода; интеграция с инструментами статического анализа типов.
- **Поддержка Java** — парсер и конвейер извлечения для второго целевого языка.
- **Альтернативные форматы вывода** — HTML, PDF или wiki наряду с Markdown.
- **Интеграция в процесс разработки** — автоматический запуск при изменении исходников в системе непрерывной интеграции.
